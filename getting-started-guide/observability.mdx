Axiom helps you leverage the power of timestamped event data. A common use case of event data is observability (o11y) in the field of software engineering. Observability is the ability to explain what’s happening inside a software system by observing it from the outside. It allows you to understand the behavior of systems based on their outputs such as telemetry data, which is a type of event data.

Software engineers most often work with timestamped event data in the form of logs, metrics, and traces. However, Axiom believes that event data reflects a much broader range of interactions, crossing boundaries from engineering to product management, security, and beyond.

## Types of event data in observability

Traditionally, observability has been associated with three pillars, each effectively a specialized view of event data:

- **Logs**: Logs record discrete events, such as error messages or access requests, typically associated with engineering or security.
- **Traces**: Traces track the path of requests through a system, capturing each step’s duration. By linking related spans within a trace, developers can identify bottlenecks and dependencies.
- **Metrics**: Metrics quantify state over time, recording data like CPU usage or user count at intervals. Product or engineering teams can then monitor and aggregate these values for performance insights.

In Axiom, these observability elements are stored as event data, allowing for fine-grained, efficient tracking across all three pillars.

## Logs, traces, and events

Axiom excels at collecting, storing, and analyzing timestamped event data.

For logs and traces, Axiom offers unparalleled efficiency and query performance. You can send logs and traces to Axiom from a wide range of popular sources. For more information, see [Send data to Axiom](/send-data/methods).

## Metrics support

Axiom provides first-class support for metrics through MetricsDB, a purpose-built metrics datastore. MetricsDB handles high-cardinality time-series data and is designed to store and query OpenTelemetry (OTel) metrics efficiently.

### MetricsDB

MetricsDB is optimized for metrics workloads in ways that a general-purpose event store is not:

- **High-cardinality tags**: Store metrics with many tag dimensions without cost penalties or performance degradation.
- **Efficient compression**: Client-side aggregation reduces data volume for high-frequency measurements.
- **Standardized OTel types**: Supports Gauge, Histogram, Sum, and Summary metric types with clear query semantics.
- **Faster queries**: Storage format is optimized for time-series visualizations without per-query processing overhead.
- **Unified observability**: Query metrics alongside logs, traces, and events using the same platform.

For a detailed comparison of MetricsDB and EventDB, see [Metrics](/query-data/metrics).

### Ingest metrics

Axiom ingests OTel metrics the same way it ingests other OTel data—via the OpenTelemetry Collector or any OTLP-compatible SDK. You must use a dedicated dataset with the **Metrics** type selected when creating it.

For more information, see [Send OpenTelemetry data to Axiom](/send-data/opentelemetry).

To get started quickly, [axiom-hostmetrics](https://github.com/axiomhq/axiom-hostmetrics) is a small CLI that streams host metrics (CPU, memory, disk, network) directly to Axiom with no full Collector setup required.

### Query metrics with MPL

Axiom includes MPL (Metrics Processing Language), a query language designed for metrics data. MPL combines the readability of APL with expressive power similar to PromQL.

A typical MPL query:

```kusto
`otel-demo-metrics`:`go.memory.used`
| where `k8s.deployment.name` == "checkout"
| align to 5m using avg
```

MPL supports filtering by tags, time-window alignment, grouping, histogram aggregations, and arithmetic across multiple metrics. If you already use PromQL, you can translate your existing queries to MPL with minimal changes.

For more information, see [Introduction to MPL](/mpl/introduction) and [Migrate PromQL queries to Axiom](/mpl/migrate-metrics).

### Dashboards and monitors

OTel metrics work in dashboards and monitors the same way as logs and traces. You can build visualizations from metrics queries, set threshold-based alerts, and combine multiple signals in a single panel.

For more information, see [Dashboards](/dashboards/overview) and [Monitors](/monitor-data/monitors).
