Use the `loggamma` function in APL to compute the natural logarithm of the absolute value of the [gamma function](https://en.wikipedia.org/wiki/Gamma_function). This is equivalent to `log(abs(gamma(x)))` but avoids the numeric overflow that occurs when `gamma(x)` itself becomes too large to represent as a floating-point number.

`loggamma` is useful when you work with large inputs to the gamma function in statistical calculations, such as log-likelihood computations, Bayesian modeling, or custom anomaly scoring. Use it whenever `gamma(x)` would overflow to infinity.

## For users of other query languages

If you come from other query languages, this section explains how to adjust your existing queries to achieve the same results in APL.

<AccordionGroup>
<Accordion title="Splunk SPL users">

Splunk SPL does not include a built-in `loggamma()` function. You typically implement it using external libraries or the Machine Learning Toolkit.

<CodeGroup>
```sql Splunk example
| eval loggamma_val = null()
````

```kusto APL equivalent
['sample-http-logs']
| extend loggamma_val = loggamma(x)
```
</CodeGroup>

</Accordion>

## Usage

### Syntax

```kusto
loggamma(x)
```

### Parameters

| Name | Type | Required | Description |
|------|------|----------|-------------|
| `x` | real | Yes | The input value. Must not be zero or a negative integer. |

### Returns

The natural logarithm of the absolute value of the gamma function of `x`.

## Use case examples

<Tabs>
<Tab title="Log analysis">

Use `loggamma` to compute a log-scale gamma weight for request durations without risking numeric overflow that would occur with `gamma` on large inputs.

**Query**

```kusto
['sample-http-logs']
| where req_duration_ms > 0
| extend duration_s = req_duration_ms / 1000.0
| extend loggamma_val = loggamma(duration_s)
| project _time, id, req_duration_ms, loggamma_val
```

[Run in Playground](https://play.axiom.co/axiom-play-qf1k/query?initForm=%7B%22apl%22%3A%22%5B%27sample-http-logs%27%5D%20%7C%20where%20req_duration_ms%20%3E%200%20%7C%20extend%20duration_s%20%3D%20req_duration_ms%20%2F%201000.0%20%7C%20extend%20loggamma_val%20%3D%20loggamma%28duration_s%29%20%7C%20project%20_time%2C%20id%2C%20req_duration_ms%2C%20loggamma_val%22%7D)

**Output**

| _time | id | req_duration_ms | loggamma_val |
|---|---|---|---|
| 2024-11-14 10:00:00 | user-1 | 1000.0 | 0.0000 |
| 2024-11-14 10:01:00 | user-2 | 2000.0 | 0.0000 |
| 2024-11-14 10:02:00 | user-3 | 10000.0 | 16.1181 |

`loggamma(1) = log(gamma(1)) = log(1) = 0`. For a 10 s request, `loggamma(10) = log(9!) ≈ 16.12`. This grows much more slowly than `gamma` itself, making it safe for large values.

</Tab>
<Tab title="OpenTelemetry traces">

Use `loggamma` to produce a stable log-gamma score from span durations, useful in log-likelihood or scoring pipelines.

**Query**

```kusto
['otel-demo-traces']
| extend duration_ms = duration / 1ms
| extend duration_s = duration_ms / 1000.0
| where duration_s > 0
| extend loggamma_val = loggamma(duration_s)
| project _time, span_id, ['service.name'], duration_ms, loggamma_val
```

**Output**

| _time | span_id | service.name | duration_ms | loggamma_val |
|---|---|---|---|---|
| 2024-11-14 10:00:00 | span-abc | checkout | 5000.0 | 3.1781 |
| 2024-11-14 10:01:00 | span-def | cart | 1000.0 | 0.0000 |

Longer spans produce higher `loggamma_val` scores. Because `loggamma` grows in log space, it remains finite even for very long spans.

</Tab>
<Tab title="Security logs">

Use `loggamma` to apply a smoothly growing weight to error-response durations in a custom risk-scoring formula.

**Query**

```kusto
['sample-http-logs']
| where status == '500' and req_duration_ms > 0
| extend duration_s = req_duration_ms / 1000.0
| extend loggamma_val = loggamma(duration_s)
| project _time, id, status, req_duration_ms, loggamma_val
| order by loggamma_val desc
```

[Run in Playground](https://play.axiom.co/axiom-play-qf1k/query?initForm=%7B%22apl%22%3A%22%5B%27sample-http-logs%27%5D%20%7C%20where%20status%20%3D%3D%20%27500%27%20and%20req_duration_ms%20%3E%200%20%7C%20extend%20duration_s%20%3D%20req_duration_ms%20%2F%201000.0%20%7C%20extend%20loggamma_val%20%3D%20loggamma%28duration_s%29%20%7C%20project%20_time%2C%20id%2C%20status%2C%20req_duration_ms%2C%20loggamma_val%20%7C%20order%20by%20loggamma_val%20desc%22%7D)

**Output**

| _time | id | status | req_duration_ms | loggamma_val |
|---|---|---|---|---|
| 2024-11-14 10:00:00 | user-9 | 500 | 9000.0 | 10.6046 |
| 2024-11-14 10:01:00 | user-5 | 500 | 3000.0 | 0.6931 |
| 2024-11-14 10:02:00 | user-3 | 500 | 1000.0 | 0.0000 |

Sorting by `loggamma_val` descending surfaces the highest-latency error events first, even when durations span many orders of magnitude.

## List of related functions

- [gamma](/apl/scalar-functions/mathematical-functions/gamma): Returns the gamma function directly. Use it for small inputs where overflow is not a concern.
- [log](/apl/scalar-functions/mathematical-functions/log): Returns the natural logarithm. Use it for general log-transformation without the gamma relationship.
- [exp](/apl/scalar-functions/mathematical-functions/exp): Returns e^x. Use it to exponentiate `loggamma` results back to the original gamma scale when the values are small enough.
- [isfinite](/apl/scalar-functions/mathematical-functions/isfinite): Returns whether a value is finite. Use it to verify that `loggamma` results are valid for downstream calculations.
- [abs](/apl/scalar-functions/mathematical-functions/abs): Returns the absolute value. Note that `loggamma(x)` equals `log(abs(gamma(x)))`, so `abs` is implicitly applied to `gamma(x)` before the log.
