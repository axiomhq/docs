Use the `sign` function in APL to return the sign of a numeric expression: +1 for positive values, 0 for zero, and -1 for negative values.

`sign` is useful when you care about the direction of a value rather than its magnitude. For example, you can use `sign` to classify whether a request latency is above or below a baseline, detect whether a metric is trending up or down, or encode deviations as ternary labels for downstream classification.

## For users of other query languages

If you come from other query languages, this section explains how to adjust your existing queries to achieve the same results in APL.

<AccordionGroup>
<Accordion title="Splunk SPL users">

Splunk SPL uses `signum()` rather than `sign()`. The semantics are identical: it returns +1, 0, or -1.

<CodeGroup>
```sql Splunk example
| eval direction = signum(deviation)
````

```kusto APL equivalent
['sample-http-logs']
| extend direction = sign(deviation)
```
</CodeGroup>

</Accordion>
<Accordion title="ANSI SQL users">

In ANSI SQL, `SIGN()` works the same as in APL: it returns +1, 0, or -1 for positive, zero, and negative inputs respectively.

<CodeGroup>
```sql SQL example
SELECT SIGN(deviation) AS direction FROM logs
```

```kusto APL equivalent
['sample-http-logs']
| extend direction = sign(deviation)
```
</CodeGroup>

</Accordion>
</AccordionGroup>

## Usage

### Syntax

```kusto
sign(x)
```

### Parameters

| Name | Type | Required | Description |
|------|------|----------|-------------|
| `x` | real | Yes | A real number. |

### Returns

- `+1` if `x` is positive.
- `0` if `x` is zero.
- `-1` if `x` is negative.

## Use case examples

<Tabs>
<Tab title="Log analysis">

Use `sign` to classify each request as faster (-1), on-target (0), or slower (+1) than a 200 ms baseline.

**Query**

```kusto
['sample-http-logs']
| extend deviation = req_duration_ms - 200.0
| extend deviation_sign = sign(deviation)
| project _time, id, req_duration_ms, deviation, deviation_sign
```

[Run in Playground](https://play.axiom.co/axiom-play-qf1k/query?initForm=%7B%22apl%22%3A%22%5B%27sample-http-logs%27%5D%20%7C%20extend%20deviation%20%3D%20req_duration_ms%20-%20200.0%20%7C%20extend%20deviation_sign%20%3D%20sign%28deviation%29%20%7C%20project%20_time%2C%20id%2C%20req_duration_ms%2C%20deviation%2C%20deviation_sign%22%7D)

**Output**

| _time | id | req_duration_ms | deviation | deviation_sign |
|---|---|---|---|---|
| 2024-11-14 10:00:00 | user-1 | 450.0 | 250.0 | 1 |
| 2024-11-14 10:01:00 | user-2 | 80.0 | -120.0 | -1 |
| 2024-11-14 10:02:00 | user-3 | 200.0 | 0.0 | 0 |

A `deviation_sign` of +1 means the request was slower than the baseline; -1 means it was faster. This ternary encoding is useful for grouping or counting by direction.

</Tab>
<Tab title="OpenTelemetry traces">

Use `sign` to label each span as slower (+1), on-target (0), or faster (-1) than a 150 ms expected latency.

**Query**

```kusto
['otel-demo-traces']
| extend duration_ms = duration / 1ms
| extend deviation = duration_ms - 150.0
| extend deviation_sign = sign(deviation)
| project _time, span_id, ['service.name'], duration_ms, deviation, deviation_sign
```

[Run in Playground](https://play.axiom.co/axiom-play-qf1k/query?initForm=%7B%22apl%22%3A%22%5B%27otel-demo-traces%27%5D%20%7C%20extend%20duration_ms%20%3D%20duration%20%2F%201ms%20%7C%20extend%20deviation%20%3D%20duration_ms%20-%20150.0%20%7C%20extend%20deviation_sign%20%3D%20sign%28deviation%29%20%7C%20project%20_time%2C%20span_id%2C%20%5B%27service.name%27%5D%2C%20duration_ms%2C%20deviation%2C%20deviation_sign%22%7D)

**Output**

| _time | span_id | service.name | duration_ms | deviation | deviation_sign |
|---|---|---|---|---|---|
| 2024-11-14 10:00:00 | span-abc | checkout | 850.0 | 700.0 | 1 |
| 2024-11-14 10:01:00 | span-def | cart | 80.0 | -70.0 | -1 |
| 2024-11-14 10:02:00 | span-ghi | frontend | 150.0 | 0.0 | 0 |

Summarizing the count of +1 and -1 spans per service tells you the ratio of slow-to-fast spans and helps identify consistently degraded services.

</Tab>
<Tab title="Security logs">

Use `sign` to flag HTTP 500 responses as occurring above or below a normal baseline duration, helping differentiate fast-fail errors from slow-timeout errors.

**Query**

```kusto
['sample-http-logs']
| where status == '500'
| extend deviation = req_duration_ms - 500.0
| extend deviation_sign = sign(deviation)
| project _time, id, status, req_duration_ms, deviation_sign
```

[Run in Playground](https://play.axiom.co/axiom-play-qf1k/query?initForm=%7B%22apl%22%3A%22%5B%27sample-http-logs%27%5D%20%7C%20where%20status%20%3D%3D%20%27500%27%20%7C%20extend%20deviation%20%3D%20req_duration_ms%20-%20500.0%20%7C%20extend%20deviation_sign%20%3D%20sign%28deviation%29%20%7C%20project%20_time%2C%20id%2C%20status%2C%20req_duration_ms%2C%20deviation_sign%22%7D)

**Output**

| _time | id | status | req_duration_ms | deviation_sign |
|---|---|---|---|---|
| 2024-11-14 10:00:00 | user-9 | 500 | 1800.0 | 1 |
| 2024-11-14 10:01:00 | user-5 | 500 | 80.0 | -1 |

Errors with `deviation_sign = +1` are slow (likely timeouts or resource exhaustion); those with -1 are fast (likely immediate failures or rate-limit responses). This distinction helps triage error categories quickly.

## List of related functions

- [abs](/apl/scalar-functions/mathematical-functions/abs): Returns the absolute value. Use it to get magnitude when direction from `sign` is not enough.
- [round](/apl/scalar-functions/mathematical-functions/round): Rounds a value. Use it to normalize values before comparing signs.
- [not](/apl/scalar-functions/mathematical-functions/not): Reverses a boolean. Use it alongside `sign` for boolean-style conditions on direction.
- [pow](/apl/scalar-functions/mathematical-functions/pow): Raises a value to a power. Use `pow(x, 2)` together with `sign(x)` to keep direction while amplifying magnitude.
- [sqrt](/apl/scalar-functions/mathematical-functions/sqrt): Returns the square root. Use it to compute magnitude, and `sign` to determine direction, when you need a signed root.
