Use the `log2` function in APL to compute the base-2 logarithm of a positive numeric value. The function is the inverse of `exp2`.

`log2` is useful when working with data measured on a binary scale, such as memory sizes, network packet lengths, or binary tree structures. Because each unit increase in `log2(x)` corresponds to a doubling of `x`, `log2` is a natural choice for analyzing values that grow or shrink by factors of two. You can also use it to compute geometric means in base-2 space.

## For users of other query languages

If you come from other query languages, this section explains how to adjust your existing queries to achieve the same results in APL.

<AccordionGroup>
<Accordion title="Splunk SPL users">

Splunk SPL doesn't include a built-in `log2()` function. You compute it as `ln(x) / ln(2)` or `log(x) / log(2)`.

<CodeGroup>
```sql Splunk example
| eval log2_duration = ln(req_duration_ms) / ln(2)
````

```kusto APL equivalent
['sample-http-logs']
| extend log2_duration = log2(req_duration_ms)
```
</CodeGroup>

</Accordion>
<Accordion title="ANSI SQL users">

Standard SQL does not define a `LOG2()` function. In SQL Server you compute it as `LOG(x) / LOG(2)`. PostgreSQL offers `LOG(2, x)` for base-2 logarithms.

<CodeGroup>
```sql SQL example
SELECT LOG(req_duration_ms) / LOG(2) AS log2_duration FROM logs
```

```kusto APL equivalent
['sample-http-logs']
| extend log2_duration = log2(req_duration_ms)
```
</CodeGroup>

</Accordion>
</AccordionGroup>

## Usage

### Syntax

```kusto
log2(x)
```

### Parameters

| Name | Type | Required | Description |
|------|------|----------|-------------|
| `x` | real | Yes | A positive real number (x > 0). |

### Returns

- The base-2 logarithm of `x`.
- `null` if `x` is negative, zero, or cannot be converted to a real value.

## Use case examples

<Tabs>
<Tab title="Log analysis">

Use `log2` to express request durations on a binary log scale, where each unit corresponds to a doubling of the duration. This is useful for bucketizing data by powers of two.

**Query**

```kusto
['sample-http-logs']
| where req_duration_ms > 0
| extend log2_duration = log2(req_duration_ms)
| project _time, id, req_duration_ms, log2_duration
```

[Run in Playground](https://play.axiom.co/axiom-play-qf1k/query?initForm=%7B%22apl%22%3A%22%5B%27sample-http-logs%27%5D%20%7C%20where%20req_duration_ms%20%3E%200%20%7C%20extend%20log2_duration%20%3D%20log2%28req_duration_ms%29%20%7C%20project%20_time%2C%20id%2C%20req_duration_ms%2C%20log2_duration%22%7D)

**Output**

| _time | id | req_duration_ms | log2_duration |
|---|---|---|---|
| 2024-11-14 10:00:00 | user-1 | 1.0 | 0.000 |
| 2024-11-14 10:01:00 | user-2 | 64.0 | 6.000 |
| 2024-11-14 10:02:00 | user-3 | 1024.0 | 10.000 |

A request of 64 ms gives `log2 = 6`; one of 1024 ms gives `log2 = 10`. Integer values correspond exactly to powers of two, making it easy to bucket durations by binary magnitudes.

</Tab>
<Tab title="OpenTelemetry traces">

Use `log2` to compute a binary-scale summary of span durations per service. Combine with `exp2` to recover the geometric mean on the original scale.

**Query**

```kusto
['otel-demo-traces']
| extend duration_ms = duration / 1ms
| where duration_ms > 0
| summarize avg_log2 = avg(log2(duration_ms)) by ['service.name'], bin(_time, 1h)
| extend geometric_mean = exp2(avg_log2)
| project _time, ['service.name'], avg_log2, geometric_mean
```

[Run in Playground](https://play.axiom.co/axiom-play-qf1k/query?initForm=%7B%22apl%22%3A%22%5B%27otel-demo-traces%27%5D%20%7C%20extend%20duration_ms%20%3D%20duration%20%2F%201ms%20%7C%20where%20duration_ms%20%3E%200%20%7C%20summarize%20avg_log2%20%3D%20avg%28log2%28duration_ms%29%29%20by%20%5B%27service.name%27%5D%2C%20bin%28_time%2C%201h%29%20%7C%20extend%20geometric_mean%20%3D%20exp2%28avg_log2%29%20%7C%20project%20_time%2C%20%5B%27service.name%27%5D%2C%20avg_log2%2C%20geometric_mean%22%7D)

**Output**

| _time | service.name | avg_log2 | geometric_mean |
|---|---|---|---|
| 2024-11-14 10:00:00 | checkout | 8.13 | 280.5 |
| 2024-11-14 10:00:00 | cart | 7.13 | 140.2 |
| 2024-11-14 10:00:00 | frontend | 5.80 | 55.8 |

A one-unit difference in `avg_log2` between services means one service is roughly twice as slow as the other.

</Tab>
<Tab title="Security logs">

Use `log2` to summarize error-response durations in binary log space per country. A higher `avg_log2` indicates that errors are exponentially slower in that region.

**Query**

```kusto
['sample-http-logs']
| where status == '500' and req_duration_ms > 0
| summarize avg_log2 = avg(log2(req_duration_ms)) by ['geo.country']
| project ['geo.country'], avg_log2
| order by avg_log2 desc
```

[Run in Playground](https://play.axiom.co/axiom-play-qf1k/query?initForm=%7B%22apl%22%3A%22%5B%27sample-http-logs%27%5D%20%7C%20where%20status%20%3D%3D%20%27500%27%20and%20req_duration_ms%20%3E%200%20%7C%20summarize%20avg_log2%20%3D%20avg%28log2%28req_duration_ms%29%29%20by%20%5B%27geo.country%27%5D%20%7C%20project%20%5B%27geo.country%27%5D%2C%20avg_log2%20%7C%20order%20by%20avg_log2%20desc%22%7D)

**Output**

| geo.country | avg_log2 |
|---|---|
| BR | 8.95 |
| DE | 7.64 |
| US | 6.99 |

Each unit increase in `avg_log2` means error responses are twice as slow on average. Brazil's errors are roughly 2^(8.95-6.99) ≈ 4× slower than those in the US.

</Tab>

## List of related functions

- [exp2](/apl/scalar-functions/mathematical-functions/exp2): Returns 2^x. Use it as the inverse of `log2`.
- [log](/apl/scalar-functions/mathematical-functions/log): Returns the natural logarithm. Use it when you prefer the natural base rather than base-2.
- [log10](/apl/scalar-functions/mathematical-functions/log10): Returns the base-10 logarithm. Use it for order-of-magnitude analysis instead of binary analysis.
- [pow](/apl/scalar-functions/mathematical-functions/pow): Raises a value to a power. Use it to compute powers of two directly as an alternative to `exp2`.
- [round](/apl/scalar-functions/mathematical-functions/round): Rounds a value. Use it after `log2` to bucket data into binary power groups.
